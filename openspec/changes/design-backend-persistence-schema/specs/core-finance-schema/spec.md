## ADDED Requirements

### Requirement: Accounts are durable financial containers
The persistence schema SHALL model accounts with durable IDs, name, balance, icon, timestamps, and a deletion policy that protects referenced transactions.

#### Scenario: Account has transactions
- **WHEN** an account is referenced by one or more transactions
- **THEN** hard deletion MUST be restricted or converted to a soft delete rather than cascading transactions silently

#### Scenario: Account is listed
- **WHEN** active accounts are listed
- **THEN** soft-deleted accounts MUST be excluded by default

### Requirement: Categories and subcategories are normalized
The persistence schema SHALL model categories and subcategories as separate relational records instead of persisting subcategories only as string arrays.

#### Scenario: Category has subcategories
- **WHEN** a category is persisted with subcategories
- **THEN** each subcategory MUST have a durable ID, category foreign key, name, and deterministic order or creation timestamp

#### Scenario: Transaction references a subcategory
- **WHEN** a transaction is assigned to a subcategory
- **THEN** it MUST reference a persisted subcategory row or carry an explicit nullable subcategory reference

### Requirement: Transactions store canonical financial values
The persistence schema SHALL model transactions with canonical amount, currency, date, type, account, optional category/subcategory, and optional source links.

#### Scenario: Regular transaction is persisted
- **WHEN** a regular transaction is created
- **THEN** it MUST store account, signed or directional Decimal amount, currency, transaction date, name, and transaction type

#### Scenario: Saving transaction is persisted
- **WHEN** a transaction has type `saving`
- **THEN** it MUST reference a goal or be rejected by the domain service

#### Scenario: Transaction is generated by cuota
- **WHEN** a transaction is generated from an installment plan
- **THEN** it MUST preserve the installment plan link and installment index/count metadata or equivalent relational fields

### Requirement: Core foreign keys preserve integrity
The persistence schema SHALL enforce foreign keys for account, category, subcategory, goal, installment plan, and investment-entry links where those links exist.

#### Scenario: Transaction references missing account
- **WHEN** a transaction write references an account that does not exist or is deleted
- **THEN** the write MUST be rejected

#### Scenario: Transaction references missing category
- **WHEN** a transaction write references a category or subcategory that does not exist or is deleted
- **THEN** the write MUST be rejected or normalized through an explicit uncategorized policy

### Requirement: Core indexes support finance queries
The persistence schema SHALL include indexes for common account, category, date, and domain-link queries.

#### Scenario: Monthly transaction list is queried
- **WHEN** transactions are filtered by date range or month
- **THEN** the schema MUST support indexed lookup by transaction date

#### Scenario: Account or category summary is queried
- **WHEN** summaries are grouped by account, category, or subcategory
- **THEN** the schema MUST support indexed lookup for those foreign keys
