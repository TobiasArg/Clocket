# feature-domain-clean-start-verification Specification

## Purpose
Verify that migrated feature domains start cleanly from backend-owned data without importing, merging, or trusting legacy browser localStorage records or IDs.

## Requirements
### Requirement: Legacy feature localStorage does not affect migrated runtime
The frontend SHALL ignore or clear legacy localStorage records for migrated feature domains during backend-owned runtime.

#### Scenario: Legacy feature localStorage keys exist
- **WHEN** the app starts with legacy budgets, goals, cuotas, investments, snapshots, refs, or settings localStorage keys present
- **THEN** migrated runtime behavior SHALL use backend data as canonical and SHALL NOT render legacy localStorage records as active data

#### Scenario: Legacy IDs exist in browser storage
- **WHEN** legacy feature-domain records contain IDs that were not generated by backend APIs
- **THEN** migrated frontend flows SHALL NOT treat those IDs as canonical backend references

### Requirement: Backend-generated IDs remain canonical
Migrated feature-domain records SHALL continue using backend-generated IDs as canonical identifiers.

#### Scenario: New migrated record is created from UI
- **WHEN** the frontend creates a migrated-domain record through an HTTP repository
- **THEN** the UI SHALL use the backend-returned ID and SHALL NOT synthesize or persist a localStorage-style ID

### Requirement: Persistence verification excludes local data merge
Stabilization SHALL verify persistence by backend read-after-write behavior rather than importing or merging localStorage records.

#### Scenario: Backend persistence is verified
- **WHEN** a migrated record is created or updated during QA
- **THEN** verification SHALL confirm backend read-after-write and browser refresh persistence without copying data into localStorage
